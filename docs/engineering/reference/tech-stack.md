# Legend Engine — Technology Stack & Key Dependencies

> **Audience:** Developers and architects who need to understand what third-party libraries  
> and external systems are involved, how versions are managed, and the rationale behind key choices.

---

## 1. Version Management Strategy

All dependency and plugin versions are declared as **properties in the root `pom.xml`**
(`groupId: org.finos.legend.engine`, `artifactId: legend-engine`). Sub-modules inherit these
properties and reference them as `${property.name}` — they do **not** declare their own versions.

This is a **Bill of Materials (BOM) pattern** without a separate BOM artifact: the root POM acts
as both parent and BOM. The `dependencyManagement` section in the root POM pins every transitive
dependency that needs a specific version override.

**Upgrading a dependency:**

1. Change the version property in the root `pom.xml`.
2. Run `mvn dependency:analyze` to detect newly unused or missing declarations.
3. Run the full test suite to verify.

**Legend dependencies** (`legend-pure`, `legend-shared`, etc.) are tracked as properties:

```xml
<legend.pure.version>5.79.0</legend.pure.version>
<legend.shared.version>0.33.0</legend.shared.version>
```

These are upgraded in lock-step when the companion repos release.

## 3. Pure Runtime: legend-pure

| Property | Value |
|----------|-------|
| Version | `5.79.0` |
| Property | `legend.pure.version` |
| Key artifact | `org.finos.legend.pure:legend-pure-m4-pct` |

The `legend-pure` library provides:

- The Pure language compiler and interpreter.
- The M3/M4 metamodel (type system, function types, multiplicities).
- The compiled-core runtime that executes Pure functions in Java via generated bytecode.
- `CodeRepository` / `CodeStorage` — the classpath-based Pure source loader.

Legend Engine bundles its own `.pure` source files as classpath resources and contributes them
to the Pure runtime as additional `CodeRepository` instances.

### 3.1 Versioning policy and upgrade procedure

`legend.pure.version` is declared in the root `pom.xml` and inherited by every module.
It must be changed in exactly **one place**.

**Version compatibility rule:** `legend-engine` and `legend-pure` are released in lock-step.
Each `legend-engine` release targets exactly one `legend-pure` version. There is no supported
mixed-version configuration. The `legend-stack-release.yml` GitHub Actions workflow coordinates
releases across both repositories.

**What breaks on a version mismatch:**

- Startup failure if the PAR binary format changes between versions (the compiled Pure runtime
  refuses to load PAR files built by a different compiler version).
- `ClassCastException` or `NoSuchMethodError` at runtime if `CompiledExecutionSupport` or the
  generated accessor API changes.
- Silent correctness regressions if the behaviour of a built-in Pure function changes.

**Upgrade procedure:**

1. Update `legend.pure.version` in the root `pom.xml`.
2. Run `mvn install -DskipTests -T 4` — build failures here indicate API breakages in
   `CompiledExecutionSupport`, `PureRuntime`, generated accessor classes, or `PureModel`.
3. Fix any compilation errors (typically updated method signatures in `CompiledExecutionSupport`
   or changed `Root_meta_*` generated class names).
4. Run `mvn test` (or at minimum the PCT and relational suites) to catch behavioural regressions.
5. Common failure patterns to look for:
   - New PAR checksum mismatch warnings at startup — regenerate PAR files by rebuilding the
     relevant `*-pure` modules.
   - PCT test failures indicating a changed function result — review the `legend-pure` changelog
     for intentional behaviour changes and update expected values accordingly.
   - `NullPointerException` in compilation — a new validation pass in `legend-pure` may now
     reject a model that was previously accepted; check `legend-pure` release notes.

## 4. Collections: Eclipse Collections

| Property | Value |
|----------|-------|
| Version | `10.2.0` |
| Property | `eclipsecollections.version` |

**Rationale:** Eclipse Collections provides richer collection APIs than the JDK (`MutableList`,
`ImmutableList`, `RichIterable`, etc.) with better performance characteristics and functional
idioms. It is used **pervasively** throughout the codebase — virtually every collection type is
from Eclipse Collections rather than `java.util`. New code should follow the same pattern.

Key types used:

- `MutableList` / `ImmutableList` — `FastList.newList()`, `Lists.immutable.with(...)`
- `RichIterable` — the base for streaming operations (`.select`, `.collect`, `.flatCollect`)
- `ConcurrentMutableMap` (via `ConcurrentHashMap`) — thread-safe maps in `PureModel`

## 5. Serialisation: Jackson

| Property | Value |
|----------|-------|
| Version | `2.10.5` / `2.10.5.1` (databind) |
| Properties | `jackson.version`, `jackson.databind.version` |

Jackson is the **primary JSON serialisation library**. Key usage:

- Protocol POJOs (`PureModelContextData`, `ExecutionPlan`, etc.) are Jackson-annotated.
- `@JsonSubTypes` / `@JsonTypeInfo` enable polymorphic deserialisation of protocol elements.
- `ObjectMapperFactory.getNewStandardObjectMapperWithPureProtocolExtensionSupports()` is the
  canonical way to get a configured `ObjectMapper` — always use this rather than `new ObjectMapper()`.
- Named subtypes for all `PackageableElement` implementations are registered at server startup
  via `Bootstrap.getObjectMapper().registerSubtypes(new NamedType(...))`.

## 7. SQL Templating: FreeMarker

| Property | Value |
|----------|-------|
| Version | `2.3.34` |
| Property | `freemarker.version` |

**Rationale:** SQL execution nodes carry parameterised SQL as FreeMarker templates. At execution
time, `FreeMarkerExecutor` resolves variables from `ExecutionState` into the template. This allows
the same plan to be re-executed with different parameter values without regeneration.

Example template in a plan:

```sql
SELECT * FROM Person WHERE id IN (${ids?join(',')})
```

`FreeMarkerConditionalExecutionNode` evaluates a FreeMarker boolean expression to choose between
two sub-plans (e.g. to switch between cache-hit and cache-miss paths).

Pac4J (`org.pac4j:pac4j-core`, `...pac4j-jaxrs`, `...pac4j-jersey`) handles HTTP-level

## 11. Parsing: ANTLR4

| Property | Value |
|----------|-------|
| Version | `4.8-1` |
| Property | `antlr.version` |

ANTLR4 is used for all grammar parsing (`.g4` files). The `maven-antlr4-plugin` generates
lexer/parser Java classes from the grammar files. Grammar files live alongside the generated
sources but are **not** tracked in `target/`.

Key grammar files (in `legend-engine-language-pure-grammar`):

- `CodeLexerGrammar.g4` / `CodeParserGrammar.g4` — top-level section parser
- `DomainParserGrammar.g4` — Pure domain (classes, functions, enums)
- `MappingParserGrammar.g4` — mapping definitions
- Each `xts-*` grammar module adds its own `.g4` files

## 12. Java Compilation at Runtime: Janino / JDK Compiler

| Library | Version property |
|---------|-----------------|
| Janino | `janino.version = 3.1.0` |

`EngineJavaCompiler` (in `legend-shared`) can compile Java source code at runtime using either
the JDK's `javax.tools.JavaCompiler` (preferred when available) or Janino as fallback.
This is used to JIT-compile the `JavaPlatformImplementation` sources embedded in execution plans.

**Important:** JDK 11 must be used (not JRE) for the `javax.tools` API to be available.

| Property | Value |
|----------|-------|

| Property | `arrow.version` |

Apache Arrow (`org.apache.arrow:arrow-vector`, `arrow-memory-netty`) is used as a streaming
result format for high-throughput columnar data transfer. The `/execute` endpoint accepts
`Accept: application/x-arrow-stream` to return results as an Arrow IPC stream instead of JSON.
